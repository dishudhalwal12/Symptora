Section 1

Why This Needed to Exist

  • My records are everywhere, so one simple health question turns into a scavenger hunt across PDFs, chats, and gallery folders.
  • My lab values look off, but a high glucose or pressure number still does not tell me whether the bigger risk picture is calm or serious.
  • Most symptom tools feel thin, because they throw out a label with no history, no context, and no path to the next question.
  • Past checks disappear fast, which means I cannot tell whether my condition is drifting, holding, or getting worse.
02

Section 2

The Product

Symptora, shipped in this repo as Medify, is a web app where a user signs in, builds a health profile, uploads records, runs disease checks, and gets a readable result with risk band, probability, major factors, and saved history. It keeps uploads, prediction outputs, and dashboard trends in one place so each check feels connected instead of isolated.

What separates it from a throwaway demo is the end-to-end loop: frontend flow, Firebase data, Python inference, explainable output, and admin monitoring all sit inside one working product.

  • Symptom Explorer - starts from what the user feels before asking for stricter clinical values.
  • Disease Modules - runs diabetes, heart, kidney, liver, and X-ray-linked assessment paths.
  • Record Locker - stores reports, prescriptions, scans, and keeps them tied to the right user.
  • Result Panel - shows probability, confidence, risk level, and the factors that pushed the output.
  • History + PDF Export - saves earlier runs and lets the user reopen or export a clean summary later.
  • Admin Control Room - tracks users, uploads, model health, and module usage.

  • Inspect system health for Firebase, ML API, and Gemini support.

Section 5

The Flow

  1. Open the landing page, choose a route, and the user sees a clear entry into symptom-first screening or account creation.

  2. Sign in or register, and the app drops the user into a private dashboard instead of a loose one-page form.

  3. Fill the health profile, so age, habits, history, and baseline values become part of the working context the user can see later.

  4. Choose a symptom route, a disease module, or the records locker, and the screen shifts from intake into focused action.

  5. Submit values or upload a file, and the frontend sends the right payload to Firebase storage paths or the ML API for inference.

  6. Review the result, where the user sees prediction label, probability, explainable factors, and a saved history trail for the next visit.

Section 7

What's Not Done Yet

  • Chest X-ray inference is not dependable in the current build because the wiring exists, but the trained image artifact is still missing in many setups.
  • Kidney support is uneven between the documents and the live app, so the product story is ahead of the strongest finished workflow.
  • Automatic report extraction is still partial, which means users may need to confirm or re-enter values after upload.
08

Section 8

If We Had More Time

  • Build OCR-backed report intake so uploaded PDFs can prefill usable medical values instead of asking the user to type them again.
  • Add full kidney-model parity and tighten every route that still promises more than the present build can prove.
  • Connect doctor-style review tools for side-by-side comparisons, trend notes, and guided follow-up checks.

Section 9

Why This Mattered

This project turned a messy health-data problem into a working full-stack product loop: sign in, save context, upload records, run ML checks, read the result, and come back later with history still intact. The real lesson was not one model or one screen. It was learning how frontend, Firebase, Python inference, and explanation layers have to agree before a product feels real. The build still has edges, but it already proves the team did more than decorate a form. Some projects end as screenshots. This one ends with a system people can actually walk through.

Built to be defended. Built to be remembered.
